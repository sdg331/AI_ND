## 1. 프로젝트 한눈에 보기 (Overview)

- **프로젝트명:** FridgeChef (내 냉장고 속 AI 셰프)
- **한 줄 소개:** 냉장고 안에 있는 재료를 최대한으로 활용하여 레시피를 창조하고, 요리의 과정을 AI가 실시간으로 솔루션을 제공한다.
- **해결하고자 하는 문제 (Problem Statement):**
    1. 재료 활용법 부재로 인하여 버려지는 식재료와 비용이 줄 수 있도록 도움
    2. 단순 레시피 서치 시 발생하는 돌발 상황에 대한 해결 방법을 AI 챗봇이 도와줌으로, 보다 완성도 있는 음식 제작 가능
- **타겟 사용자 (Persona):**
    - **김지금 (23세, 대학생 자취러):**  자취를 하기에 주머니 사정이 넉넉하지 않은 상황이다. 집에 있는 재료와 기구를 최대한으로 활용하여 식비를 줄이고자 함
- **기획 배경 (Why Now?):** 기존의 '검색형' 레시피 앱은 수만 가지 재료의 조합(Combination)을 모두 DB화할 수 없었습니다. 이제 **생성형 AI(Generative AI)**의 등장으로, 사전에 정의되지 않은 레시피도 실시간으로 '추론'하고 '창조'할 수 있게 되었습니다.

## 3. 사용자 조사 및 인사이트 (User Research)

- **조사 방식:** 기존 레시피 앱(만개의레시피, 쿡패드 등) 리뷰 500건 분석 (문헌분석법)
- **핵심 Pain Point:**
    - "냉장고 파먹기 메뉴를 검색하면, 꼭 내가 **없는 재료(예: 파슬리, 치킨스톡)를 요구**해서 포기하게 돼요."
- **AI 적용 필요성 (Key Insights):**
    1. **검색의 한계:** 사용자가 가진 재료의 교집합을 찾는 쿼리 방식은 결과가 '0건'이거나 '부정확'할 확률이 높음. → **AI의 '생성' 능력이 필수.**
    2. **맥락의 이해:** "지금 간이 짜다"는 말에 "물 넣어"가 아닌, "양파를 더 넣어 단맛으로 중화해"라는 **맥락적 솔루션**이 필요. → **LLM의 추론 능력 필요.**
    3. **입력의 귀찮음:** 재료를 일일이 타이핑하는 건 가장 큰 이탈 요인. → **Vision AI(멀티모달) 도입 필수.**

## 4. 솔루션 구조 (Solution Architecture)

### (1) 전체 서비스 흐름

### 1. To-Be 사용자 여정 (User Journey Map)

프로토타입(v1.0) 기준, 사용자가 앱에 진입하여 요리를 완성하기까지의 핵심 흐름입니다.

1. **재료 등록 (Input Phase)**:
    - 사용자가 앱을 실행하고 **[냉장고 탭]**에 진입합니다.
    - '사진 찍기' 버튼을 눌러 식재료를 촬영하면, Vision AI가 자동으로 재료명과 수량을 인식하여 리스트에 추가합니다. (또는 텍스트로 직접 입력)
2. **재고 관리 (Management Phase)**:
    - 등록된 재료는 카테고리별(육류, 채소 등)로 자동 분류되어 구글 시트(DB)에 실시간 저장됩니다.
    - 중복된 재료가 입력될 경우, 별도 항목으로 생성되지 않고 기존 수량에 합산되어(Update) 관리 효율을 높입니다.
3. **레시피 생성 (Generation Phase)**:
    - *[AI 셰프 탭]**으로 이동하여 '레시피 추천받기' 버튼을 클릭합니다.
    - 시스템은 현재 냉장고 재료와 사용자 프로필(알레르기, 도구)을 종합하여 최적의 메뉴와 조리법을 생성합니다.
4. **결과 확인 (Action Phase)**:
    - 생성된 레시피 카드(요리명, 필요 재료, 단계별 가이드, 팁)를 확인하고 요리를 시작합니다.

### 2. 기능 구성도 (Functional Architecture)

핵심 기능 3가지를 중심으로 한 시스템의 논리적 구조입니다.

- **① 스마트 냉장고 모듈 (Ingredient Manager):**
    - Vision API 기반 식재료 이미지 인식 및 속성(이름/수량/분류) 추출.
    - 재료 데이터 CRUD (Create, Read, Update, Delete) 및 수량 자동 합산 로직.
- **② AI 셰프 엔진 (Recipe Generator):**
    - **Prompt Engineering:** 재료+프로필 데이터를 결합한 동적 프롬프트 생성.
    - **Model Routing:** 가용 가능한 AI 모델(Gemini 1.5 Flash vs Pro)을 자동으로 탐색하고 연결하는 안정성 확보 로직.
- **③ 데이터 동기화 시스템 (Data Sync):**
    - `gspread` 라이브러리를 활용하여 Frontend의 변경 사항을 Google Sheets와 실시간 양방향 동기화.

### (2) 생성형 AI 도구 활용 방식

본 프로젝트는 기획부터 구현까지 생성형 AI를 적극적으로 도입하여 개발 생산성을 극대화하였습니다.

### 1. Coding & Text AI (활용 도구: Google Gemini)

- **핵심 로직 구현 및 디버깅:**
    - Python Flask 백엔드 서버의 골격 생성 및 RESTful API 엔드포인트 설계.
    - 특히, 외부 API 연동 시 발생하는 다양한 오류(404 Model Not Found, 415 Media Type, 429 Quota Exceeded)를 해결하기 위해 **'모델 자동 감지 및 Fallback(우회) 알고리즘'**을 코딩 AI와 협업하여 구현함.
- **코드 리팩토링:**
    - 초기 복잡했던 코드를 기능별(재료 관리, 비전, 레시피) 함수로 모듈화하고, 중복 코드를 제거하여 유지보수성을 높임.

### 2. UI/UX & Asset AI (활용 도구: LLM Code Generation)

- **UI 와이어프레임 및 디자인 구현:**
    - 별도의 디자인 툴(Figma 등)을 거치지 않고, LLM에게 **"모던하고 깔끔한 카드형 UI와 하단 탭바가 있는 반응형 웹 디자인"**을 프롬프트로 요청하여 HTML/CSS 코드를 직접 생성함.
    - 사용자 경험(UX)을 고려하여 로딩 스피너(Spinner), 직관적인 아이콘, 카테고리 뱃지 디자인을 AI가 제안하고 적용함.

### 3. Backend & Workflow Automation (활용 도구: Python Logic Automation)

- **외부 API 연동 자동화 (Workflow Orchestration):**
    - 복잡한 라이브러리 의존성을 제거하고, Python의 `requests` 모듈을 사용하여 구글의 생성형 AI 서버와 직접 통신하는 **경량화된 백엔드 워크플로**를 구축함.
    - 이미지 데이터를 Base64로 인코딩하여 텍스트 프롬프트와 함께 전송하는 멀티모달 데이터 처리 파이프라인을 자동화함.
- **데이터 처리 로직:**
    - AI가 비정형 자연어(JSON 텍스트)로 응답한 데이터를 파싱(Parsing)하여, 정형 데이터(엑셀 행/열)로 변환하고 DB에 적재하는 과정을 자동화함.

## 5. 핵심 기능 시연 (Demo Description)

## 6. 기술 구조 및 AI 모델 설계

### 6.1 사용한 생성형 AI 도구 및 API

본 프로젝트는 Google의 최신 생성형 AI 모델을 핵심 엔진으로 사용하여, 텍스트 생성과 이미지 인식(Vision) 기능을 구현하였습니다.

- **핵심 AI 엔진:** Google Gemini API (Generative Language API)
- **활용 모델:**
    - **Gemini 1.5 Flash / Gemini Pro:** 사용자의 냉장고 재료와 프로필 데이터를 분석하여 요리 레시피를 생성하는 텍스트 추론 모델로 사용.
    - **Gemini Pro Vision:** 사용자가 촬영한 식재료 사진을 분석하여 재료명, 수량, 카테고리를 추출하는 멀티모달(Multi-modal) 모델로 사용.
- **연동 방식:** Python `requests` 라이브러리를 활용한 RESTful API 직접 호출 방식 (라이브러리 의존성 문제 해결 및 안정적인 통신 구현).

### 6.2 모델 응답 구조 및 프롬프트 설계

AI가 일관된 데이터를 출력하여 애플리케이션에서 오류 없이 처리할 수 있도록 **시스템 프롬프트(System Prompt)**와 **JSON 출력 형식(Output Schema)**을 엄격하게 설계하였습니다.

### (1) AI 셰프 (레시피 생성) 프롬프트

사용자의 '보유 재료'와 '개인 프로필(알레르기, 도구)'을 입력받아 맞춤형 레시피를 생성합니다.

- **입력(Prompt Input):**
    
    > "재료: [계란(3개), 양파(1개), 우유], 사용자 정보: [알레르기: 없음, 도구: 에어프라이어]. 위 재료를 메인으로 사용하여 만들 수 있는 요리 레시피 1개를 추천해줘."
    > 
- **제약 조건(Constraints):**
    
    > "응답은 반드시 프로그래밍 가능한 JSON 형식으로만 대답해."
    > 
- **출력 설계(JSON Structure):**JSON
    
    `{
      "recipeName": "에어프라이어 계란 양파 구이",
      "materialsUsed": "계란 2개, 양파 반 개",
      "cookingSteps": [
        "1. 양파를 채 썹니다.",
        "2. 계란을 풉니다.",
        "3. 에어프라이어 용기에 담아 180도에서 10분 굽습니다."
      ],
      "tip": "우유를 조금 넣으면 더 부드러워져요!"
    }`
    

### (2) 비전 인식 (재료 등록) 프롬프트

사진 데이터를 Base64 포맷으로 변환하여 전송하며, 이미지 속 객체를 식별하고 분류합니다.

- **입력(Prompt Input):** 이미지 데이터 + 텍스트 명령어
    
    > "이 사진에 있는 식재료 이름과 수량, 그리고 카테고리(육류/채소/과일 등)를 추론해줘."
    > 
- **출력 설계(JSON Structure):**JSON
    
    `{
      "name": "사과",
      "quantity": "2개",
      "category": "과일"
    }`
    

### 6.3 데이터 흐름도 (Data Flow Architecture)

서비스의 전체적인 데이터 흐름은 **[사용자 입력 → Flask 서버 처리 → AI 추론 → DB 저장 → 결과 출력]**의 순환 구조를 가집니다.

1. **입력 (Input):**
    - 사용자가 웹(Frontend)에서 재료를 텍스트로 입력하거나 사진을 업로드합니다.
    - HTML/JS가 데이터를 `FormData` 또는 `JSON` 형식으로 패키징하여 백엔드로 전송합니다.
2. **처리 (Processing - Backend & AI):**
    - **Python Flask 서버**가 요청을 수신합니다.
    - **이미지 처리:** 업로드된 이미지를 AI가 읽을 수 있는 `Base64` 포맷으로 인코딩합니다.
    - **API 호출:** 사전에 정의된 프롬프트와 데이터를 결합하여 **Google Gemini API**에 전송합니다.
    - **응답 파싱:** AI가 보낸 텍스트 응답에서 순수 JSON 데이터만 추출하고 파싱(Parsing)하여 Python 딕셔너리로 변환합니다.
3. **저장 (Storage):**
    - 가공된 데이터(재료 목록, 레시피 결과)는 **Google Sheets API (`gspread`)**를 통해 클라우드 스프레드시트에 실시간으로 기록됩니다. (CRUD 구현)
    - 동일한 재료가 입력될 경우, 기존 수량을 파악하여 자동으로 합산하는 로직이 수행됩니다.
4. **출력 (Output):**
    - 최종 처리된 데이터가 프론트엔드로 반환되어, 사용자의 화면에 '재료 목록' 또는 '레시피 카드' 형태로 시각화됩니다.

## 7. 프로젝트 결과 & 성과 (Outcome)

- **AI 활용 인사이트 (Lesson Learned):**
    1. AI는 '창조'의 도구**:** 없는 정보를 찾아주는 게 아니라, 주어진 조건으로 최선의 답을 만들어내는 LLM의 본질을 확인.
    2. 할루시네이션 제어: 존재하지 않는 요리를 만드는 것을 막기 위해 "검증된 조리법 기반"이라는 제약 조건을 거는 노하우 습득.
    3. AI의 설명에 따라 사진 인식(Vision AI ), API연동 등을 구현할 수 있음.

## 8. 한계점 및 개선 방향 (Roadmap)

- **한계점:**
    - **Vision 정확도:** 겹쳐져 있는 재료나 불투명 용기 속 재료 인식 실패율 존재.
    - **Hallucination:** 간혹 맛이 보장되지 않는 괴식 레시피 생성 가능성.
    - API키 만료ㅇ
    - 다음 버전에서 추가할 기능: 챗봇, 영수증 스캔, 커뮤니티
- **개선 방향 (Next Step):**
    - **RAG (검색 증강 생성):** 검증된 레시피 데이터베이스(만개의레시피 등)를 Vector DB로 구축하여, 생성 시 참고하도록 개선.
    - **커뮤니티 피드백 루프:** 사용자가 "맛있어요"한 레시피를 학습 데이터로 재활용(RLHF).
